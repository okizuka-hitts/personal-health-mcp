## 要対応（実装前に決定が必要）

### 1. 初回 OAuth フロー（初期認証）が未定義

`.env` に `HEALTH_PLANET_REFRESH_TOKEN` を設定するまでの手順が仕様に含まれていない。
`HEALTH_PLANET_REDIRECT_URI=http://localhost:8080/callback` が .env にあることから Authorization Code Flow を想定しているが、**初回認証を行うCLIコマンドまたはスクリプト**が必要。

**提案:** `server.py` とは別に `python -m health_mcp.tanita.auth` で起動するワンショット認証スクリプトをスコープに加える。README の「初回認証手順」に対応するコードが必要。

### 2. リフレッシュトークン更新後の `.env` 書き戻し設計が未定義

アクセストークンの自動更新（リフレッシュ）が走ると、`HEALTH_PLANET_ACCESS_TOKEN` と `HEALTH_PLANET_TOKEN_EXPIRES_AT` を更新する必要がある。`.env` ファイルへの書き戻しはコード実装が必要だが、その方針が記載されていない。

**提案:** `.env` の直接書き換えか、別途 `.tokens` ファイルを用意するかを決める。個人用途なら `.env` 書き換えで十分だが、方針を要件に明記する必要がある。

### 3. `get_profile` がプロフィールのみ必要なときの `innerscan` 呼び出し範囲

T-04 の処理フローで「キャッシュがない場合は `innerscan` エンドポイントを呼び出す」とあるが、`innerscan` には `from`・`to` パラメータが必要。**プロフィール取得目的の最小リクエスト日付範囲**が未定義。

**提案:** 「今日の0時〜23時59分59秒」などの最小範囲を仕様に明記する（測定データが0件でもレスポンスのトップレベルにプロフィール情報は返るという前提を確認すること）。

## 軽微な指摘・確認事項

### 4. プロフィールのキャッシュ手動リフレッシュ手段がない

「TTL なし・手動リフレッシュのみ」と明記されているが、`profile` テーブルを削除する手段（CLIコマンド、または `reset_profile_cache` MCP ツール）が提供されていない。運用上どう対処するか一言記載があると完成度が上がる。

### 5. `get_measurements_range` にキャッシュ更新ロジックがない（仕様確認）

T-03 は SQLite を直接参照するだけで API 呼び出しを行わない。これは意図的と思われるが、初回起動時にデータが0件の場合 `total_count: 0` が返り、ユーザーが混乱する可能性がある。仕様通りで問題なければそのまま進める。

### 6. Windows ネイティブ Python での `asyncio` イベントループポリシー

Windows では `asyncio` のデフォルトループが `ProactorEventLoop` になるため、`httpx` 非同期クライアント使用時に問題が起きる場合がある。FastMCP がこれを吸収するか確認が必要（吸収しない場合は `server.py` で `asyncio.set_event_loop_policy(asyncio.WindowsSelectorEventLoopPolicy())` が必要）。

### 7. `--verbose` の実現方法

「`--verbose` フラグ時に詳細ログ」とあるが、MCP サーバーは `claude_desktop_config.json` で起動されるため、CLIフラグより**環境変数（例: `LOG_LEVEL=DEBUG` を `.env` に追加）** のほうが設定しやすい。

### 8. `get_body_composition` のキャッシュ更新スコープ

「最終同期から1時間超 → API 呼び出し」とあるとき、呼び出し範囲は**リクエストの `start_date`〜`end_date`** か、それとも**直近30日など固定範囲**か未定義。ユーザーが過去データ（例: 半年前）を要求した場合の挙動を明確にする必要がある。
