## v1.3 新規指摘事項

### 1.【軽微】T-01 が「直近30日」固定のため、測定が30日以上空くと "No data available" になる

T-01（`get_latest_body_composition`）の処理フローが「直近30日を対象範囲として差分同期」と固定されている。旅行・入院・測定忘れ等で最終測定日が30日以上前になると、キャッシュ済みのデータがあっても同期対象外となり、結果的に `"error": "No data available"` が返る可能性がある。

**提案:** 次のいずれかを仕様に明記する。

- 案A（シンプル）: 同期後に SQLite に1件も存在しない場合は、対象範囲を90日に自動拡張して再同期する
- 案B（割り切り）: 「直近30日に測定がない場合は `"error"` を返す」と仕様として明記し、ユーザーが `get_body_composition(start_date=...)` で明示指定する

個人用途であれば案Bの方が実装がシンプル。

### 2.【軽微】`--reset-profile` が `auth.py` の責務を超えている

`python -m health_mcp.tanita.auth --reset-profile` は、SQLite の `profile` テーブルを削除する処理であり、認証（`auth.py`）ではなくストレージ（`storage/sqlite.py`）の操作になる。`auth.py` が `storage/` に依存する形になる。

個人用途のシングルスクリプトとして許容範囲ではあるが、将来的に `auth.py` と `sqlite.py` を分離してテストしたい場合に依存関係が問題になる可能性がある。

**提案:** 許容するなら「`auth.py` は `storage/sqlite.py` に依存する」と設計方針に一言添える。気になる場合は `python -m health_mcp storage --reset-profile` のような別エントリに切り出す。
