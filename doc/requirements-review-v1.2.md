## v1.2 新規指摘事項

### 1.【要確認】T-01 のキャッシュ更新ロジックと差分同期設計が矛盾している

T-01（`get_latest_body_composition`）の処理フローに「最終同期から1時間以上 → API 呼び出し」と残っているが、セクション5のキャッシュ更新方針は「差分同期（未同期範囲のみ呼び出し）」に変更されている。v1.0 の単純な TTL ベースの記述が T-01 に残ったままになっている。

**提案:** T-01 の処理フローを「差分同期ロジックに従いキャッシュ更新を判定する（セクション5参照）」に書き換え、記述を一本化する。また `CACHE_TTL_SECONDS` は差分同期設計では当日の再同期間隔に使うのか、それとも不要になったのかを明確にする。

### 2.【軽微】当日データの連続呼び出し時の再同期間隔が未定義

「`end_date` が今日を含む場合は当日分を常に再同期する」とあるが、同日中に MCP ツールを複数回呼び出した場合、毎回 API を呼ぶことになる。`CACHE_TTL_SECONDS` の用途がここに残るのか判断が必要。

**提案:** 「当日分も `CACHE_TTL_SECONDS`（デフォルト3600秒）以内の再呼び出しはキャッシュを使う」など最小限の制限を設けると過剰な API 呼び出しを防げる。仕様として確定しなくてもよいが、`CACHE_TTL_SECONDS` の役割を明記する。

### 3.【軽微】`python -m health_mcp.tanita.auth` は `[project.scripts]` ではなく `__main__.py` で実現

受け入れ条件に「`python -m health_mcp.tanita.auth` が `[project.scripts]` に登録されているか」とあるが、`python -m パッケージ` 形式の実行は `[project.scripts]` ではなく `tanita/__main__.py`（または `auth.py` 内の `if __name__ == "__main__"` ブロック）で実現する。`[project.scripts]` は `health-mcp = "health_mcp.server:main"` 形式の名前付き CLI コマンド用。

**提案:** 受け入れ条件のチェック項目を「`health_mcp/tanita/__main__.py`（または `auth.py` の `__main__` ブロック）が存在するか」に修正する。

### 4.【軽微】長期稼働プロセスでの `.env` 書き戻しとメモリ上のトークン状態

MCP サーバーは長期稼働プロセスのため、`python-dotenv` の `set_key()` で `.env` を更新してもプロセスの環境変数（`os.environ`）には反映されない。トークン更新後はファイル書き戻しと同時に **メモリ上の変数も更新する**実装が必要。

**提案:** `auth.py` のトークン更新関数が `.env` 書き戻しとともに戻り値として新トークンを返し、呼び出し元がインメモリ状態を更新する設計を明示するか、トークン管理クラスで一元管理する旨を実装メモとして記載する。
